Module 5: Memory Management

Understand address binding, paging, and memory allocation strategies

1. Introduction to Memory Management

Motivation of Memory Management

To optimize system performance, memory must be strategically managed and shared among multiple processes. Key reasons include:

  • CPU Sharing & Scheduling: To implement effective CPU scheduling, several processes must reside in memory simultaneously.
  • Performance Optimization: Sharing memory directly increases the degree of multiprogramming.
  • Management Algorithms: Specialized memory management algorithms are essential to allocate, track, and protect memory spaces cleanly.

What is Memory?

It consists of a large array of words or bytes, each assigned its own unique physical address. The CPU fetches instructions and data from memory according to the Program Counter (PC), executing code and writing back updated values.

The Lifecycle of a Process in Memory

  1. Input Queue: A collection of processes residing on disk waiting to be brought into main memory.
  2. Process Placement: A program must be loaded into memory and placed within an active process environment to execute.
  3. CPU Scheduling: The CPU Scheduler selects ready processes from memory to execute on the CPU.
  4. Execution Access: While executing, the process dynamically accesses instructions and data directly from memory.
  5. Termination: When the process finishes, its allocated memory space is declared available for other jobs.

2. Address Binding & Processing Stages

Addresses in a source program are typically symbolic. Before execution, they must be mapped to physical memory addresses through a multi-step translation chain:

Three Stages of Address Binding

Address binding can take place at different operational stages:

  • Compile Time: Compiler generates absolute code. Requires recompilation if starting address changes.
  • Load Time: Compiler generates relocatable code; final binding handled by the loader at load time.
  • Execution Time (Run Time): Binding delayed until runtime; requires hardware support for address maps (e.g., base and limit registers).

3. Logical vs. Physical Address Space

Logical (Virtual) Address: An address generated by the CPU. Programs use logical addresses and do not see physical addresses directly.

Physical Address: The actual physical memory location recognized by the hardware.

Memory-Management Unit (MMU)

The MMU translates virtual addresses into physical addresses at runtime.

Dynamic Relocation Scheme: The value stored in the relocation register (base register) is dynamically added to every logical address generated by a user process when accessing memory.

Physical Address (PA) = Logical Address (LA) + Relocation Register (RA)

Example: If RA = 14000 and LA = 346, then PA = 346 + 14000 = 14346.

4. Operational Concepts: Loading, Linking, & Overlays

Dynamic Loading

Routines are kept on disk and are not loaded into main memory until explicitly called. Advantage: improves memory utilization because unused code is never loaded.

Static vs. Dynamic Linking

Static Linking: Library routines are combined into the program executable, producing duplicate copies in memory.

Dynamic Linking: Linking postponed until run-time using stubs; library routines are shared in memory across processes.

Overlays

Keep only the necessary instructions/data in memory at any time by overwriting inactive modules. Useful when process size exceeds available memory; implemented at program level.

5. Swapping

A process can be swapped out of main memory to a backing store and brought back later.

  • Backing Store: High-speed disk large enough to hold copies of memory images for users.
  • Roll Out / Roll In: Variant where lower-priority processes are swapped out to run higher-priority jobs.

Swapping Constraints: Processes must be idle before swapping; never swap a process with pending I/O unless I/O uses OS buffers.

Sample Problem: Swapping Calculations

Problem: Assume a user job is 20,000 words, backing store latency is 8 ms, transfer rate is 250,000 words/sec. Compute total swap time out and in.

Transfer Time = 20,000 / 250,000 = 0.08 s = 80 ms

One-way Swap Time = Transfer Time + Latency = 80 ms + 8 ms = 88 ms

Total Swap Time (Out and In) = 88 ms × 2 = 176 ms

6. Memory Allocation & Protection Strategies

Modern memory is partitioned into:

  • Resident OS: Typically low memory.
  • User Processes: Typically high memory.

Hardware Address Protection

Use Base (Relocation) register and Limit register to enforce address bounds. If logical address ≥ limit → trap.

7. Contiguous Memory Allocation Schemes

Contiguous allocation requires a process to occupy a single continuous block of memory.

A. Monoprogramming (Single-Partition Allocation)

Only one user process executes at a time; memory divided between OS and single user program.

B. Multiprogramming with a Fixed Number of Tasks (MFT)

Memory divided into fixed regions; each region holds one process. Loading may use absolute or relocatable translation.

C. Multiprogramming with a Variable Number of Tasks (MVT)

Memory boundaries are dynamic; OS manages a free-space list of holes and allocates exact-sized blocks, merging adjacent holes when reclaimed.

8. Memory Fragmentation & Allocation Policies

Internal Fragmentation: Unused memory inside an allocated partition. External Fragmentation: Free memory exists but split into non-contiguous holes.

Allocation Policies

  • First Fit: Allocate first sufficiently large hole found.
  • Best Fit: Allocate smallest hole that fits (scans entire list).
  • Worst Fit / Best Available Fit: Allocate largest hole to leave large remaining block.

Solutions: Compaction (costly) and Non-contiguous Allocation (paging).

9. Paging: Non-Contiguous Memory Management

Paging eliminates external fragmentation by dividing logical address space and physical memory into fixed-size pages and frames.

Address Translation

The logical address is split into Page Number p and Page Offset d. The page table maps p to frame number f, and the physical address is computed as:

Physical Address (PA) = (Frame Number f × Page Size P) + Page Offset d

The Frame Table

The OS maintains a Frame Table with one entry per physical frame, tracking allocation and which process/page occupies it.

Fragmentation in Paging

External fragmentation: eliminated. Internal fragmentation: may occur in the last page if process size isn't a multiple of page size. Example: page size 16 KB, job 30 KB → allocated 32 KB → internal fragmentation 2 KB.
